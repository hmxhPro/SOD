
你现在是一个资深的 **AI 全栈工程师 + 计算机视觉工程师**，请帮我从零开发一个 **基于 FastAPI 的视频目标检测 Agent 系统**。
系统核心方案采用：

* **后端**：FastAPI
* **检测模型**：Grounding DINO 或 Florence-2（二选一，优先选择更适合“自然语言指定目标 + 输出检测框”的方案）
* **跟踪器**：ByteTrack
* **前端**：任选一个你最熟悉且适合快速落地的方案（推荐 React + Vite）
* **部署环境**：Linux + NVIDIA GPU
* **目标**：用户上传视频，通过自然语言描述要检测的物体，系统逐帧检测并实时回传检测结果

请直接为我生成一个**可运行、结构清晰、工程化完整**的项目。

## 一、系统功能要求

### 1. 视频上传 + 自然语言输入

用户可以在前端：

* 上传一个视频文件（如 mp4）
* 输入自然语言检测目标，例如：

  * “帮我检测视频中的菜园”
  * “帮我检测岸边的钓鱼台”
  * “帮我检测靠近水边的种菜区域”

后端接收后，自动调用视觉检测模型完成逐帧检测。

---

### 2. 检测框样式要求

检测后的图像必须画出**类似示例图example_pro.png**的效果：

* 在目标区域外画出清晰的矩形框
* 在框的上方显示目标名称文本
* 文本内容要显示用户描述对应的目标名称
* 绘制效果要美观清晰，适合前端展示
* 建议在框旁边增加置信度（可选）
* 框和文字要避免遮挡主体，风格参考常见目标检测可视化效果

输出效果示例要求类似：

* 黄色或高对比度边框
* 框上方显示“临水菜园”这种标签
* 标签背景色和字体颜色要有明显对比，便于阅读

---

### 3. 实时检测结果展示

系统不能等全部检测结束后再一次性展示，而是要**边检测边显示**。

具体要求：

* 前端在任务开始后，实时显示检测进度
* 每处理完一帧，就把这一帧的检测结果图返回给前端显示
* 前端页面上要能持续刷新最新检测结果
* 建议使用 **WebSocket** 或 **Server-Sent Events（SSE）**
* 前端要显示：

  * 当前处理到第几帧
  * 当前帧在视频中的时间点
  * 当前检测结果图

---

### 4. 每帧结果必须带时间点信息

每一帧检测结果必须附带该帧在视频中的时间点信息，例如：

* 帧序号：125
* 时间点：00:00:05.000

要求：

* 后端保存每帧对应的时间戳
* 前端实时显示这张图对应的视频时间点
* 保存到文件名中也要体现时间点，例如：

  * `frame_000125_00-00-05-000.jpg`

---

### 5. 结果文件保存与下载

检测完成后，系统必须支持下载所有检测结果帧的压缩包。

要求：

* 把所有已绘制检测框的图片保存下来
* 每张图片包含目标框、标签、时间点信息
* 所有结果图打包为一个 zip 文件
* 前端提供“下载结果 zip 包”按钮
* zip 中建议包含：

  * 全部结果图片
  * 一个 `results.json` 或 `results.csv`，记录每帧的检测信息
* 记录信息至少包括：

  * frame_id
  * timestamp
  * detected_label
  * bbox 坐标
  * score

* `POST /api/upload`
* `POST /api/detect`
* `GET /api/task/{task_id}`
* `GET /api/stream/{task_id}`
* `GET /api/download/{task_id}`

或者你认为更合理的 REST/WebSocket 结构。

需要包含：

* 创建检测任务

---

### 4. 检测与跟踪流程要求

请采用合理的视频处理流程，例如：

1. 读取视频
2. 抽取帧
3. 基于用户自然语言调用 Grounding DINO 或 Florence-2 检测目标
4. 使用 ByteTrack 做跟踪，减少重复检测开销
5. 将检测框和标签绘制到每一帧图像上
6. 保存结果图
7. 实时推送到前端
8. 检测结束后生成 zip 包

请你根据“速度优先于精度”的目标，对流程做合理优化，例如：

* 不是每一帧都全量检测
* 检测帧 + 跟踪帧结合
* 保证整体结果连贯性
* 保证框的显示尽量稳定
